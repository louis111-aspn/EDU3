PART 4. ABAP Programming Legacy — Extension Model Axis

Modification Culture

LA07~LA10이 SAP 내부 설계의 흔적이었다면, LA11은 그 설계 위에서 기업이 시스템을 확장하면서 축적한 Organizational Technical Debt를 다룬다 — SAP Technical Debt의 가장 큰 원인이다.
Origin Architecture Decision
Limited Extension Framework → Direct Modification Culture

11.1 Legacy Origin — 왜 직접 수정이 유일한 방법이었는가

초기 SAP 시스템은 대부분의 ABAP Source Code를 고객이 직접 열람하고 편집할 수 있는 형태로 제공했다. Black-box로 컴파일된 소프트웨어와 달리, 표준 로직이 기업 요구사항과 맞지 않을 때 개발자가 취할 수 있는 선택지는 단 하나 — 표준 Program/Include를 직접 여는 것이었다. 아직 구조화된 Extension Framework(사전에 정의된 확장 지점)가 존재하지 않았기 때문이다.

11.2 Technical Mechanism — Direct Modification

개발자는 SE38/SE80 같은 도구로 SAP 표준 Program을 직접 열어 로직을 수정했다. 이후 SAP가 새 버전을 출시할 때, 이 수정 내역은 표준 Object와 충돌한다 — 업그레이드 시마다 개발자가 자신이 수정한 부분을 새 표준 코드에 수동으로 다시 반영해야 했다(Modification Adjustment, 이후 SPDD/SPAU Transaction으로 도구화됨).

11.3 Why It Worked — 단기적 합리성

Direct Modification은 표준 기능과 실제 업무 요구 사이의 격차를 가장 빠르게 메우는 방법이었다. SAP가 별도 Enhancement를 내놓을 때까지 기다릴 필요 없이, Source가 열려 있다는 조건 위에서 원하는 어떤 로직이든 즉시 구현할 수 있었다. 당시 Extension Framework 자체가 존재하지 않았다는 점을 고려하면, 이는 개발자의 나쁜 습관이 아니라 유일하게 가능했던 Business Choice였다.

11.4 Modern Debt — Upgrade Risk의 누적

영역문제
Upgrade Risk 매 Release마다 수정된 Object를 수동으로 재조정(SPAU)해야 함 — Object 수가 늘어날수록 작업량이 누적됨
Regression 부담 수정 범위가 넓어질수록 Upgrade 시 Test 범위도 함께 늘어남
Core/Custom 경계 불명확 어디까지가 SAP 표준이고 어디부터가 자사 수정분인지 구분이 흐려져, "무엇을 건드려도 안전한지" 판단이 어려워짐
Architecture Misconception Warning Modification은 개발자의 잘못이 아니다. 당시 Extension Mechanism의 한계를 보완하기 위한 Business Choice였다. 문제는 이 방식이 누적되면서 SAP 표준 Core와 기업 고유 로직의 경계가 흐려졌다는 점이다.

11.5 Evolution — User Exit → BAdI → Side-by-side → Clean Core

Modification 표준 Program을 직접 수정
↓
User Exit 표준 코드 안에 미리 준비된 빈 Subroutine(CMOD/SMOD)에 로직 삽입 — 표준 자체는 건드리지 않음
↓
BAdI Object 기반 확장 — 여러 Implementation과 Filter 조건 지원
↓
Enhancement Framework Explicit/Implicit Enhancement Point로 확장 지점을 표준화
↓
Side-by-side Extension BTP 등 Core 시스템 밖에서 API로 연결되는 확장 — Core 자체는 전혀 건드리지 않음
↓
Clean Core Core ERP는 수정하지 않고, 모든 Custom Logic을 Extension Layer로 분리하는 전략 원칙 — Customization을 없애는 것이 아니라, Custom Capability가 SAP Core의 Lifecycle(Upgrade 등)을 방해하지 않도록 Boundary를 관리하는 전략이다

이 흐름은 "확장을 점점 어렵게 만든 것"이 아니라, 확장 지점을 Core로부터 점진적으로 분리해 Upgrade Risk를 줄여온 과정이다.

Architecture Reading Point LA11은 Extension Model 축의 핵심 사례이자 Part 6의 중심이다. 다음 챕터인 LA12(Transport-based Delivery)는 이 Modification/Extension 이력이 어떻게 System 간에 이동·배포되었는지, 그리고 이것이 Git 중심 현대 개발 방식과 왜 달랐는지를 다룬다.

One Sentence Takeaway
Modification Culture는 Extension Framework가 성숙하지 않았던 시대에 Business Requirement를 충족하기 위한 현실적인 선택이었다. SAP Architecture는 Modification을 부정하는 것이 아니라, Core와 Extension의 Boundary를 분리하는 방향으로 진화해왔다.
