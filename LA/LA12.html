PART 4. ABAP Programming Legacy — Delivery Governance Axis

Transport-based Delivery

LA11이 "무엇을 어떻게 확장했는가"였다면, LA12는 "그 변경을 어떻게 통제하고 운영했는가"다. 핵심 질문은 왜 SAP 개발 Lifecycle이 Git 중심 개발과 다른 방식으로 발전했는가다.
Origin Architecture Decision
Centralized Enterprise Control → Transport Management Model

12.1 Legacy Origin — 왜 중앙 통제형 배포가 필요했는가

LA04에서 본 MANDT Client 구조는 하나의 물리적 시스템 안에 여러 논리적 구역(DEV/QAS/PRD)을 두는 방식으로도 확장되었다. 문제는 이렇게 분리된 System/Client 사이에서 개발된 변경 사항을 어떻게 안전하게, 순서대로, 감사 가능하게(Auditable) 옮길 것인가였다. Enterprise ERP는 재무·법적 요구사항 때문에 임의의 직접 배포를 허용할 수 없었고, SAP는 이를 중앙 통제형 Transport Management Model로 해결했다.

12.2 Technical Mechanism — CTS / Transport Request

변경 사항은 CTS(Change and Transport System)를 통해 Transport Request 단위로 관리된다. DEV에서 만들어진 Transport Request는 Task와 Object 목록을 담고 있으며, Release된 후 QAS → PRD 순서로 순차 Import된다. Object는 변경 내역(Diff)이 아니라 전체 버전이 통째로 이동하는 방식이다.

12.3 Why It Worked — 규제된 환경에서의 거버넌스

이 모델은 개발/테스트/운영 담당자가 분리되어야 하는 규제 요구사항(예: 직무 분리, SOX 통제)이 강한 Enterprise ERP 환경에 잘 맞았다. 하나의 System of Record를 통해 어떤 변경이 언제, 누구에 의해, 어떤 순서로 이동했는지 전체 이력이 명확하게 남는다는 점이 핵심 장점이었다.

12.4 Modern Debt — Git과의 격차

영역격차
Branching/Merging Transport Request는 단일 순차 Chain으로 이동 — Git처럼 여러 Branch를 병렬로 개발하고 병합하는 모델을 기본 지원하지 않음
Version History Object 단위 전체 교체 방식이라, 코드 레벨의 세밀한 변경 이력(Diff)을 표준적으로 추적하기 어려움
병렬 개발 여러 개발자가 같은 Object를 동시에 수정할 경우 Transport 순서와 충돌 관리에 수작업이 필요
Architecture Misconception Warning Transport 기반 개발은 실패한 DevOps가 아니다. 규제된 Enterprise ERP 환경에서 변경을 중앙 통제하고 감사 가능하게 만들기 위한 합리적인 Governance Model이었다. Git과 다르게 발전한 것은 열등해서가 아니라, 애초에 다른 문제(분산 협업이 아니라 통제된 배포)를 해결하도록 설계되었기 때문이다.

12.5 Modern Evolution — abapGit / CI/CD / Cloud ALM

  • abapGit — ABAP Object를 Git 저장소로 직렬화해, Branch/Merge/Diff 등 Git 워크플로우를 ABAP 개발에 도입
  • CI/CD Pipeline 연동 — Transport 이동 단계를 자동화된 Pipeline(빌드·테스트·배포)에 통합
  • Cloud ALM — S/4HANA Cloud 환경에서 Application Lifecycle을 통합 관리

이 변화는 Transport 개념 자체를 제거한 것이 아니라, System 간 통제된 이동이라는 본래 목적을 유지하면서 그 위에 Git 스타일의 협업·이력 관리를 결합하는 방향이다.

12.6 Part 4 Closing — Architecture Summary

LA07~LA12는 각각 독립된 기술 주제가 아니라, 하나의 Development Lifecycle을 이룬다 — 개발 단위 (LA07) → 데이터 구조(LA08) → UI 결합(LA09) → 사용자 진입 방식(LA10) → 확장/수정 문화(LA11) → 배포 통제(LA12)로 이어지는 흐름이다.
