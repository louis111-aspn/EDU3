13.1 Modern Mechanism — Data Model First

LA05에서 본 것처럼, ECC 시대 ABAP 개발자는 BKPF/BSEG 같은 여러 Table을 직접 Join·집계하는 코드를 작성했다. 흔히 이 변화를 "CDS View를 쓴다"는 구현 차원으로 이해하지만, 더 정확한 표현은 Data Model First라는 설계 철학의 전환이다. CDS는 그 철학을 구현하는 수단일 뿐이다.

" ECC — ABAP가 SQL을 감싸고 Table을 직접 다룸
ABAP
  ↓
SQL
  ↓
Table

" S/4HANA — Business Model이 먼저 정의되고, CDS/ABAP는 그 아래에서 동작
Business Model
  ↓
CDS
  ↓
ABAP

계산 로직을 ABAP Loop가 아니라 View 정의와 HANA Push-down으로 표현하는 것은 이 순서 반전의 결과다 — 원인이 아니다.

" 과거: ABAP에서 직접 Loop 돌며 집계
LOOP AT it_bseg INTO ls_bseg.
  lv_total = lv_total + ls_bseg-dmbtr.
ENDLOOP.

" 현재: CDS View로 집계를 선언 -- 계산은 HANA로 Push-down
DEFINE VIEW ZI_SalesTotal AS SELECT FROM acdoca {
  key rbukrs,
  sum( hsl ) as total_amount
} GROUP BY rbukrs

13.2 Why It Solves Today's Constraints — Redundancy 제거가 먼저다

S/4HANA의 가장 큰 목적은 흔히 말하는 "실시간성"이 아니라 Redundancy 제거다. Real-time은 그 결과로 따라오는 것이지, 목적 자체가 아니다.

" ECC — 같은 데이터를 여러 형태로 중복 저장
Index Table
Aggregate Table
Summary Table
Application Cache

" S/4HANA — 하나의 진실 지점 위에서 필요한 View를 즉시 계산
Memory (HANA)
  └─ ACDOCA
       └─ CDS

LA06에서 본 Batch 중심 운영은 익일 데이터를 전제로 했다. 하지만 그 근본 원인은 "느려서"가 아니라 FI/CO/AA 등 하위 원장이 나중에 서로 맞춰봐야 하는(Reconciliation) 구조였기 때문이다. Single Source of Truth(ACDOCA)가 먼저 확보되고, 그 위에서 Real-time 조회가 자연스럽게 뒤따른다.

13.3 New Trade-offs — 새로운 학습 곡선과 호환성 비용

영역Trade-off
Modeling 학습 곡선 Annotation 중심 CDS 모델링은 단순 ABAP Report 작성보다 진입 장벽이 높음
호환성 계층 과거 Aggregate Table을 참조하던 Z-Program을 위해 Compatibility View를 유지해야 하는 경우가 있음
성능 튜닝 관점 전환 "야간 Batch Job 최적화"에서 "CDS View/SQL Push-down 최적화"로 개발자의 성능 사고 방식 자체가 바뀜
Database Dependency 과거 ABAP는 DB를 추상화해 감췄지만, 이제 개발자가 Execution Plan·Join·Cardinality· Annotation·Push-down을 이해해야 함 — DB를 몰라도 되던 시대가 끝났다
Architecture Misconception Warning S/4HANA는 ECC를 최적화한 제품이 아니라, ERP의 계산 위치를 다시 정의한 아키텍처다. 성능 개선은 결과일 뿐이며, 본질은 데이터 중심 아키텍처(Data Model First)로의 전환이다. ABAP 개발자의 역할도 계산 로직 작성에서 Semantic Data Model 설계로 이동했다.

13.4 Next Evolution — Digital Core가 여는 다음 질문

Digital Core가 "하나의 진실 지점"을 전제로 하려면, 그 Core 자체가 흔들리지 않아야 한다 — 즉 LA11에서 본 Modification 문화가 계속되면 Digital Core의 전제가 무너진다. 다만 이는 Customization의 시대가 끝난다는 뜻이 아니다. Customization은 사라지는 것이 아니라, 그 Boundary가 재정의되는 것이다 — 다음 챕터인 LA14(Clean Core)는 바로 이 지점에서 출발한다.

Architecture Reading Point LA13은 Part 5의 출발점이다. LA05(Header-Item→ACDOCA)와 LA06(Batch→Real-time)에서 이미 다룬 Architecture 변화가, 이제 "ABAP 개발자가 무엇을 짜는가"라는 관점에서 재해석된다.

Architectural Principle
S/4HANA ABAP은 ERP를 대체한 것이 아니라, ABAP 개발의 기본 단위를 Table에서 CDS View로, 개발자의 역할을 계산 로직 작성자에서 Semantic Data Model 설계자로 전환시켰다.
Design Law
Move computation closer to the data, not the data closer to the computation.
