20.2 Why It Solves — LA01으로 돌아가기

이 시리즈는 R/3에서 시작했다. 그 여정을 한 줄로 그리면 다음과 같다.

R/3
  ↓
Function Group → Header Line → Dynpro → Modification → Transport
  ↓
Clean Core → Contract → Business Object → Enterprise

Function Group의 Global Variable도, Dynpro의 Screen-Logic 결합도, Modification 문화도 — 그 어느 것도 실수가 아니었다. None of these were mistakes. They were architectural responses to different constraints. Clean Core, Contract, Business Object, Enterprise 역시 마찬가지다 — 이들도 오늘의 제약에 대한 Architectural Response일 뿐이다.

20.3 Architectural Judgment — Trade-off가 아니라 판단

  • No universal architecture — 모든 상황에 맞는 단 하나의 정답은 없다
  • Constraints always change — 제약은 항상 변하고, 그에 따라 설계도 변해야 한다
  • Principles outlive implementations — 구현은 낡지만 원칙은 남는다
  • Context determines design — 같은 원칙도 맥락에 따라 다른 구현으로 나타난다
Architecture Misconception Warning Legacy is not obsolete architecture.

It is architecture shaped by yesterday's constraints.

Modern architecture succeeds only when it understands why legacy existed.
