4.1 Legacy Origin — 하나의 시스템, 여러 Business Environment

과거에는 하나의 SAP 시스템(하나의 Application Server 집합, 하나의 Database)을 구축하는 비용 자체가 매우 컸다. 그런데 기업은 대개 하나의 사업만 운영하지 않는다 — 본사와 자회사, 실 운영 환경과 교육/테스트 환경, 여러 국가 법인이 동시에 존재한다. 사업 단위마다 별도의 물리적 시스템을 두는 것은 당시로서는 감당하기 어려운 비용이었다.

SAP의 해법은 하나의 물리적 시스템 위에, 여러 개의 논리적 Client(Mandant)를 두는 것이었다. 같은 프로그램·같은 테이블 구조를 공유하면서도, 데이터는 Client 단위로 완전히 분리되도록 설계했다.

하나의 물리적 SAP System (하나의 DB, 하나의 Application Server 집합)
Client 100운영 (Production)
Client 200테스트 (QA)
Client 300교육/샌드박스

4.2 Technical Mechanism — MANDT as Key Field

이 논리적 분리는 Database Table 설계 차원에서 구현된다. Customizing, Master Data, Transaction Data를 담는 Client-dependent Table은 예외 없이 MANDT를 Table의 첫 번째 Primary Key Field로 갖는다.

" Client-dependent Table의 전형적인 Key 구조
KNA1 (Customer Master)
  MANDT   " Client — 항상 첫 번째 Key
  KUNNR   " Customer Number

" 반면 Client-independent Table (예: ABAP Repository 객체)은
" MANDT 없이 전체 System에서 공유된다
Important Concept 모든 SAP Table이 MANDT를 갖는 것은 아니다. ABAP Dictionary는 Client-dependent Object(대부분의 Transaction Data, 일부 Configuration Data)와 Cross-client Object(일부 Repository Object, System-wide Control Data)를 구분해서 관리한다. MANDT는 전자에만 적용된다.

사용자가 SAP GUI에 로그인할 때 Client(100/200/300)를 먼저 선택하는 이유가 여기 있다 — 로그인 Session에 Client가 고정되면, SAP Runtime Environment는 이 Client Context를 기반으로 Client-dependent Data Access를 제어한다. Open SQL 경로에서는 이 제어가 일관되게 적용되지만, Native SQL이나 일부 직접 DB 접근 경로에서는 동일하게 보장되지 않는다는 점은 유의할 부분이다.

4.3 현대 문제 — Cloud Tenant Architecture와의 충돌

이 모델은 "하나의 시스템, 여러 논리적 구역"이라는 전제 위에 서 있다. 문제는 Cloud SaaS 시대의 Multi-tenancy가 요구하는 격리 수준이 다르다는 점이다.

영역격차
격리 단위 MANDT는 같은 DB/코드 위의 Row 단위 격리. Cloud Multi-tenancy는 Tenant별 독립적인 Infrastructure·Scaling·보안 경계를 전제로 함
External Query 복잡화 SAP Kernel을 거치지 않는 Raw SQL/외부 도구 접근에서는 MANDT 조건을 개발자가 직접 챙겨야 함
데이터 접근 제약 Client 간 데이터 이동(Client Copy 등)은 SAP 표준 절차를 거쳐야 하며, 임의의 Cross-Client Query는 설계상 지원 대상이 아님
Architecture Misconception Warning MANDT는 Cloud Multi-tenancy의 "실패한 버전"이 아니다. MANDT는 초기 Enterprise ERP 환경에서 하나의 시스템 자원을 여러 Business Environment가 공유하기 위한 합리적인 Isolation Strategy였다. 현대 Cloud Architecture에서는 Isolation Boundary가 Application Layer에서 Infrastructure/Tenant Layer로 이동하면서 그 역할이 변화하였다.

4.4 S/4HANA / BTP 시대 대응 방향

S/4HANA에서도 MANDT 구조 자체는 기술적으로 유지된다. 다만 Public Cloud/RISE with SAP와 같은 현대 배포 모델에서는, 실질적인 격리 경계가 Client Level이 아니라 System 또는 Tenant Level로 이동한다 — 즉 고객마다 별도의 System Instance를 할당하는 방식이 Cloud Multi-tenancy의 기본 전제에 더 가깝다.

  • MANDT는 여전히 존재하지만, Cloud 환경에서는 "여러 사업부를 하나의 System 안에 욱여넣는 수단"이 아니라 System 내부의 운영/테스트 구역 분리 등 제한된 용도로 쓰이는 경우가 많음
  • Tenant 간 완전한 격리가 필요한 경우, MANDT가 아니라 별도 System/Tenant 배치가 아키텍처 결정 사항이 됨
Legacy Isolation Layer
Client
  └─ System

Modern Isolation Layer
Tenant
  └─ Identity
       └─ Service Boundary

즉 현대 Cloud Architecture로의 이동은 단순히 "Client를 없애는 것"이 아니라, 격리 책임이 Identity Management / Authorization / Service Isolation / Cloud Tenant Governance 영역으로 옮겨가는 과정이다.

Architecture Reading Point LA04는 LA01의 Logical Isolation 축을 여는 첫 사례다. LA02·LA03(Data Representation 축)과는 다른 갈래이며, 다음 챕터인 LA05(Header-Item Data Model)는 다시 Database Architecture 축으로 이동한다.
One Sentence Takeaway
MANDT는 Legacy Debt가 아니라 시대가 요구했던 Isolation Strategy이며, SAP Architecture의 변화는 데이터를 나누는 방식에서 Business Boundary를 정의하는 방식으로 진화한 과정이다.
