PART 2. 데이터 Representation의 역사와 한계

Leading Zero (ALPHA Conversion)

Storage가 절대적으로 비쌌던 시대, SAP는 Key를 고정 길이 숫자 문자열로 저장하는 방식을 선택했다. 이 챕터는 그 선택이 오늘날 왜 Integration Debt가 되었는지를 다룬다.
Origin Architecture Decision (LA01 연결)
Storage Optimization → Leading Zero

2.1 Legacy Origin — 왜 Leading Zero가 필요했는가

과거 Database Index 구조는 가변 길이 Key를 다루는 데 취약했다. Leading Zero의 핵심 목적은 "Storage를 아끼는 것" 자체가 아니라, 고정 길이 Character Key 구조를 유지하면서 Sorting, Indexing, Comparison 처리를 단순화하는 데 있었다.

  • Fixed Length Key — 가변 길이 처리 로직 없이 Key를 다룰 수 있음
  • Lexical Sorting — 문자열 정렬만으로 숫자 순서가 그대로 보장됨
  • Consistent Comparison — Key 비교 시 길이·타입 보정 로직이 불필요

즉 12345를 저장 공간 때문에 0000012345로 만든 것이 아니라, SAP 내부 Key 체계에서 Fixed Length Key + Lexical Sorting + Consistent Comparison을 동시에 확보하기 위한 설계였다. Leading Zero는 "숫자를 예쁘게 보이기 위한 표시 규칙"이 아니라, Database Layer의 물리적 제약에 대한 Application Layer의 대응이었다.

2.2 Internal / External Format — SAP Key Representation 구조

이 대응은 SAP ABAP Dictionary에 "하나의 값이 두 가지 표현을 가진다"는 구조로 남았다. 사용자가 화면에서 보는 값(External)과 Database에 저장되는 값(Internal)이 분리된다.

External Format (화면 / 입력) 12345
↔ ALPHA Routine
Internal Format (Database 저장) 0000012345
Important Concept SAP Database는 "사용자가 보는 값"이 아니라 "SAP Internal Format"을 기준으로 Key를 관리한다. 이 원리는 Customer 하나에 국한되지 않고, Material Number, Vendor, GL Account, WBS Element 등 NUMC/CHAR 계열 Key 전반에 동일하게 적용되는 공통 원리다.

이 변환은 임의로 발생하지 않는다. 해당 필드가 참조하는 Domain(예: 고객 번호 Domain)에 ALPHA라는 Conversion Routine이 지정되어 있을 때만 작동한다. Domain의 Data Type이 CHAR 또는 NUMC이고 Conversion Routine이 붙어 있는 필드에서, 값의 입출력 시점마다 자동으로 이 변환이 개입한다.

2.3 CONVERSION_EXIT_ALPHA 메커니즘

이 변환을 실제로 수행하는 것은 Function Module 쌍이다.

" 화면 입력 → Database 저장 (외부 → 내부)
CALL FUNCTION 'CONVERSION_EXIT_ALPHA_INPUT'
  EXPORTING
    input  = '12345'
  IMPORTING
    output = lv_internal.       " '0000012345'

" Database 조회 → 화면 표시 (내부 → 외부)
CALL FUNCTION 'CONVERSION_EXIT_ALPHA_OUTPUT'
  EXPORTING
    input  = '0000012345'
  IMPORTING
    output = lv_external.       " '12345'
Routine 목적
CONVERSION_EXIT_ALPHA_INPUT External → Internal 변환
CONVERSION_EXIT_ALPHA_OUTPUT Internal → External 변환

Dynpro 화면, Standard Report, 그리고 대부분의 표준 ABAP 개발 패턴에서는 이 변환이 Framework 차원에서 자동으로 호출된다. 다만 Conversion Exit는 Database Trigger가 아니라 SAP Application Layer의 Formatting Rule이라는 점이 핵심이다 — 즉 이 관례를 거치지 않고 Database에 직접 접근하는 모든 경로에서는 변환이 보장되지 않는다.

2.4 현대 Integration 시대의 문제

ALPHA Conversion은 SAP GUI와 표준 ABAP 실행 경로 안에서는 문제없이 작동했다. 문제는 시스템이 API, BI Tool, 외부 Interface와 직접 연결되기 시작하면서 드러났다.

연결 경로 변환 책임이 SAP 밖으로 노출될 때의 영향
외부 API ↔ SAP External API가 12345를 그대로 전달할 경우, 수신 측에서 Internal Format(0000012345)과의 값 해석 차이를 별도로 처리해야 함
BI / Reporting Tool Raw Table을 직접 조회할 경우 Leading Zero가 그대로 노출되어 데이터 해석 비용이 늘어남
SQL 직접 조회 WHERE kunnr = '12345'는 Internal Format 기준으로 재작성해야 의도한 결과를 얻을 수 있음
개발자 작업 부담 Interface·Migration 개발마다 ALPHA 변환 로직을 수동으로 다시 구현해야 함

여기서 핵심 문제는 ALPHA Conversion 자체가 아니다. SAP 표준 API/OData 경로에서는 이미 Conversion이 처리되는 경우가 많다. 진짜 문제는 SAP 내부의 Semantic Rule이 Integration Boundary 밖으로 나가는 순간 사라지는 현상이다.

Architecture Misconception Warning Leading Zero 자체는 Legacy Debt가 아니다. ALPHA Conversion은 SAP GUI/ABAP 표준 경로 안에서는 지금도 정상적으로 동작한다. 진짜 문제는 SAP 내부 Key Representation Rule이 외부 Integration Boundary까지 동일한 Semantic Model로 전달되지 못했던 Architecture Gap이다.

2.5 S/4HANA / API 시대 대응 방향

S/4HANA와 CDS 기반 개발에서도 ALPHA Conversion Routine 자체는 여전히 존재한다. 다만 대응 방향은 "규칙을 없애는 것"이 아니라 "규칙을 Application Layer 관례에서 Semantic Layer(CDS)로 끌어올려 일관되게 적용되도록 만드는 것"에 가깝다.

Legacy Table Access
        ↓
Semantic Layer
        ↓
CDS View
        ↓
API Consumer
  • CDS View 및 OData 서비스 단에서 Conversion Routine을 명시적으로 적용해 외부로 노출되는 값의 일관성을 확보하는 방향으로 설계
  • API 설계 시 "Internal Key 그대로 노출" vs "External Format 변환 후 노출"을 아키텍처 결정 사항으로 명시하는 것이 실무 원칙 — 이 판단을 누락하면 동일한 문제가 API 계층에서 재발한다

다만 여기서 과장하지 말아야 할 지점이 있다: CDS가 Leading Zero 문제를 "완전히 해결"하는 것은 아니다. 현대 SAP Architecture의 방향은 Conversion Logic을 제거하는 것이 아니라, Semantic Layer에서 일관되게 관리하는 쪽에 가깝다.

Architecture Reading Point LA02는 LA01에서 정의한 Storage Optimization → Leading Zero Decision의 첫 번째 실사례다. 같은 Storage/Data Representation 축의 다음 사례는 LA03(Currency Decimal Shift)이며, Isolation 축의 사례는 LA04(MANDT Client)로 이어진다.
One Sentence Takeaway
Legacy Data Format은 문제가 아니라, 과거 최적화 규칙을 현재 Architecture Boundary 안에서 어떻게 계승할 것인가가 핵심이다.
