PART 4. ABAP Programming Legacy — Programming Paradigm Axis

Internal Table Header Line

LA07이 "상태(State)를 어떻게 관리했는가"였다면, LA08은 "데이터 구조와 반복 처리를 어떻게 단순화하려 했는가"다. 변수 선언을 줄이는 편의성이 어떻게 유지보수 비용으로 전환되었는지를 다룬다.
Origin Architecture Decision
Memory Simplification → Implicit Data Container Pattern

8.1 Legacy Origin — 선언을 줄이기 위한 설계

Internal Table(반복 데이터를 담는 ABAP 자료구조)을 다루려면 원래 두 개의 객체가 필요하다 — 전체 데이터를 담는 Table Body와, 한 줄(Row)을 담는 Work Area다. 초기 ABAP은 이 둘을 하나의 선언으로 합칠 수 있는 WITH HEADER LINE 구문을 제공했다. 하나의 식별자가 Table 전체와 그 안의 한 행을 동시에 가리키게 함으로써, 개발자가 매번 별도의 Work Area 변수를 선언하지 않아도 되게 만든 것이다.

8.2 Technical Mechanism — WITH HEADER LINE / OCCURS

" Legacy 선언 — 하나의 이름이 Table과 Row를 동시에 가리킴
DATA: it_data OCCURS 0 WITH HEADER LINE.

it_data-field1 = 'A'.     " Header Line(현재 행)에 값 대입
APPEND it_data.           " Header Line 내용을 Table Body에 추가

LOOP AT it_data.          " 별도 INTO 절 없이 Header Line에 자동으로 채워짐
  WRITE it_data-field1.
ENDLOOP.

OCCURS n과 WITH HEADER LINE은 역사적으로 함께 쓰였지만 개념적으로는 서로 다른 것이다. OCCURS는 Table의 초기 Memory 확보 크기를 지정하던 구형 Internal Table 선언 방식이고, WITH HEADER LINE은 Table과 Row를 하나의 이름으로 묶는 Implicit Work Area 기능이다. 둘 다 구형 선언 패턴에 속하지만 서로 다른 목적을 가진 구문이 함께 관례적으로 사용된 것이다. it_data라는 하나의 이름이 문맥에 따라 "Table 전체"를 의미하기도 하고 "지금 이 한 행"을 의미하기도 한다.

8.3 Why It Worked — 화면 필드 바인딩과의 궁합

당시 Dynpro 화면 개발에서는 화면 필드와 ABAP 변수 이름이 직접 대응되는 경우가 많았다. Table과 Row를 하나의 이름으로 합쳐두면, 화면 로직·반복 처리·조회 로직 전반에서 선언해야 할 이름의 수가 줄어들고 코드가 짧아졌다. 제한된 개발 생산성 환경에서 "선언을 줄이는 것"은 실질적인 이점이었다.

8.4 Modern Debt — 모호성과 유지보수 비용

영역문제
Semantic Ambiguity it_data가 코드 위치에 따라 "전체 Table"인지 "현재 한 행"인지 한눈에 구분되지 않음
Implicit Data Flow LOOP AT it_data.처럼 INTO 절이 생략되면, 데이터가 어디로 채워지는지 코드만 보고는 알기 어려움
Modularization 제약 Header Line은 선언과 강하게 결합되어 있어, Row 하나만 독립적으로 함수에 전달하거나 재사용하기 어려움
Architecture Misconception Warning Internal Table Header Line은 개발자가 편의를 위해 만든 비정상적 사용 방식이 아니다. 초기 ABAP Programming Model에서 선언 복잡도를 줄이기 위한 합리적인 선택이었다. 문제는 명시적인 Data Model과 Modular Design이 중요해진 시대에 같은 패턴을 그대로 유지하면서 발생했다.

8.5 Modern Evolution — Explicit Work Area / Field Symbol / Inline Declaration

" Modern ABAP — Table과 Work Area를 명시적으로 분리
DATA: lt_data TYPE STANDARD TABLE OF ty_data,
      ls_data TYPE ty_data.

LOOP AT lt_data INTO ls_data.
  ...
ENDLOOP.

" Field Symbol — Copy 없이 참조 (성능 이점)
LOOP AT lt_data ASSIGNING FIELD-SYMBOL().
  -field1 = 'A'.
ENDLOOP.

" Constructor Expression — 선언과 동시에 값 채움
DATA(lt_result) = VALUE ty_table( ( field1 = 'A' ) ( field1 = 'B' ) ).

Explicit Work Area는 Table과 Row를 분리해 모호성을 없애고, Field Symbol은 Copy 없는 참조로 성능과 명시성을 동시에 확보한다. Inline Declaration과 Constructor Expression(VALUE)은 선언과 데이터 구성을 한 문장으로 명확하게 표현한다 — 선언을 줄이려던 원래 목적은 유지하되, 모호성 없이 달성하는 방향이다.

One Sentence Takeaway
Internal Table Header Line은 초기 ABAP 환경에서 Data Declaration과 Processing Context를 단순화하기 위한 Programming Model이었다. 현대 ABAP Architecture는 같은 생산성을 유지하면서도 Explicit Data Structure와 Predictable Behavior를 중심으로 진화하였다.
