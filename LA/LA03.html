3.1 Legacy Origin — 왜 이런 저장 방식이 필요했는가

SAP는 하나의 Amount Field Type으로 전 세계 모든 통화를 관리해야 했다. 문제는 통화마다 소수점 자리수가 다르다는 점이다 — KRW·JPY는 소수점이 없고, USD·EUR은 2자리, 일부 통화는 3자리를 쓴다. SAP가 선택한 방식은 통화의 의미 자체를 통일한 것이 아니라, Amount Field의 저장 구조는 하나로 표준화하고, 통화별 Decimal Precision 차이는 Currency Metadata로 관리하는 것이었다. 이것이 Data Representation Standardization이 Currency 영역에서 구체화된 결과다.

3.2 Technical Storage Format — Internal Value 구조

Database에 저장되는 Amount 값은 통화의 실제 소수점 자리수와 무관하게, 내부적으로는 2-decimal 기준 구조를 따른다. 화면에 보이는 값과 저장된 값 사이의 차이는 그 통화의 Decimal Place 메타데이터를 알아야만 해석할 수 있다.

DB Internal Value 10.00
↔ Decimal Place
Metadata (KRW = 0)
Display Value 1,000 KRW

KRW는 소수점 자리수가 0이기 때문에, 내부 저장값 10.00은 실제로 1,000을 의미한다. 이 변환은 Amount Field 자체가 아니라, 그 값과 함께 전달되는 Currency Code가 있어야만 성립한다.

Important Concept DB에 저장된 숫자는 독립적인 Business Amount가 아니다. Currency Key와 함께 해석되어야 하는 Technical Value다.

3.3 Currency Master Semantics — Decimal Place 메타데이터

SAP는 기본적으로 통화를 2-decimal로 가정한다. 2-decimal이 아닌 통화만 별도 마스터 테이블 (TCURX)에 예외로 등록된다.

통화Decimal Places비고
USD / EUR2 (Default)TCURX에 별도 등록 불필요
KRW / JPY0TCURX 예외 등록 대상
BHD 등 일부 통화3TCURX 예외 등록 대상

즉 TCURX는 "모든 통화의 소수점 정보"가 아니라, 기본값(2-decimal)과 다른 통화만 예외로 관리하는 테이블이다. 이 예외 처리 구조 자체가 Data Representation Standardization의 산물이다 — 표준을 하나로 통일하고, 예외만 별도로 관리하는 방식.

3.4 Financial Amount Meaning — Semantic Loss의 위험

여기서부터가 LA02와 결정적으로 다른 지점이다. Amount 값은 숫자 하나만으로는 아무 의미가 없다. 10.00이라는 값은 Currency Code 없이는 "10.00원"인지 "1,000원"인지 "10.00달러"인지 알 수 없다. 즉 SAP의 Amount Field는 Technical Storage Format(숫자)과 Business Meaning(실제 금액)이 분리되어 있고, 이 둘은 Currency Code라는 다리를 통해서만 다시 결합된다.

이 다리가 끊어지는 순간 — 즉 Amount 값이 Currency Code 없이, 또는 잘못된 Decimal Place 가정과 함께 전달되는 순간 — 흔히 "100배 오류"라고 부르는 현상이 나타난다. 다만 정확히 말하면 Decimal Shift 로직 자체가 틀린 것이 아니라, Currency Metadata Context가 유실되면서 발생하는 Semantic Loss다.

Architecture Misconception Warning Currency Decimal Shift는 금액 계산 오류가 아니다. 문제는 Amount 값이 그 값을 해석하는 데 필요한 Currency Semantic Context(통화 코드, Decimal Place 메타데이터)와 분리된 채 전달될 때 발생하는 관리의 문제다.

3.5 External Reporting Boundary — 현대 Integration 시대의 문제

이 Semantic Loss는 SAP 내부에서는 거의 발생하지 않는다. SAP 표준 화면과 Report는 항상 Amount와 Currency Code를 Pair로 다루도록 설계되어 있기 때문이다. 문제는 이 Pair 구조가 SAP 밖으로 나가는 경계에서 깨질 때다.

3.6 S/4HANA / API 시대 대응 방향

현대 SAP Architecture의 방향은 Amount 필드를 Currency Code와 분리 불가능한 Semantic Unit으로 다루는 것이다. CDS 모델링에서 Amount 필드에 대응하는 Currency Code 필드를 명시적으로 연결해 두면, 이후 소비되는 모든 계층(API, Analytics)에서 이 Pair 관계가 유지된다.

  • Amount 필드는 반드시 Currency Code 필드와 Pair로 노출되도록 API/CDS 설계 시 명시
  • 외부 시스템에 Raw Internal Value를 그대로 넘기기보다, Display Format으로 변환된 값을 넘길지 여부를 아키텍처 결정 사항으로 명확히 할 것

다만 이 역시 "문제를 제거"하는 것이 아니라 Semantic Pair 관계를 계층 전체에서 계승시키는 방향이라는 점은 LA02와 동일하다.

Architecture Reading Point LA02(Leading Zero)와 LA03(Decimal Shift)은 모두 Data Representation Standardization이라는 같은 Decision에서 나온 자매 사례다. 다음 챕터인 LA04(MANDT Client)는 Storage/Representation 축이 아니라 Logical Isolation 축으로 이동한다.
One Sentence Takeaway
Financial Data의 핵심은 숫자가 아니라 의미이며, 현대 SAP Architecture는 Amount와 Currency Context를 하나의 Business Semantic으로 관리하는 방향으로 진화하였다.
