LA05가 "무엇을 어떻게 나눠 저장했는가"의 문제였다면, LA06은 "언제 계산했는가"의 문제다. Batch는 낡은 방식이 아니라, 제한된 Computing Resource 시대의 합리적인 Scheduling Strategy였다.

6.1 Legacy Origin — 왜 Batch 중심 운영이 필요했는가

LA05에서 본 Header-Item 구조는 대량의 Line Item을 실시간으로 집계하는 데 근본적인 비용을 수반한다. 과거 Computing Resource(CPU, Memory, 동시 처리 능력)로는 업무 시간 중 전체 데이터에 대한 실시간 집계·계산을 감당할 수 없었다. SAP의 해법은 무거운 계산 작업을 업무 외 시간(야간, 주말)에 몰아서 처리하는 것이었다 — Batch Window라는 개념이다.

주간 업무 시간Transaction 입력 집중
→
야간 Batch Window집계·마감·평가 Job 실행
→
익일 Report전일 마감 데이터 기준

6.2 Technical Mechanism — Background Processing 구조

SAP는 이 Scheduling 전략을 Background Job이라는 개념으로 구현했다. Job은 정해진 시각 또는 이벤트에 맞춰 실행되도록 예약(Variant + Schedule)되고, 실행 결과는 Spool/Log로 남는다.

  • 기간말 마감(Period-end Closing) Program — 감가상각, 원가배부, 외화평가 등
  • Currency Valuation Run — 결산 시점의 환율 재평가
  • Allocation Run — Cost Center/Profit Center 간 배부
  • Summary/Aggregate Table 갱신 — 다음 날 Report가 참조할 집계 Table을 야간에 미리 계산

Background Job은 단순 자동 실행 기능이 아니다. 제한된 Resource 환경에서 Processing Order와 Business Control(마감 순서, 계산 선후관계, 실행 결과 검증)을 관리하기 위한 Operational Architecture였다.

6.3 Data Availability Model — Batch Window와 Data Freshness

이 구조에서는 Report가 참조하는 데이터가 항상 "어제 밤 마감 시점"을 기준으로 한다. 실시간 Raw Transaction Table을 직접 조회할 수는 있지만, 집계·평가가 반영된 Reporting용 데이터는 Batch가 완료되어야 비로소 이용 가능하다. 이는 일관된 시점 스냅샷(Point-in-Time Consistency)과 실시간성(Freshness) 사이의 Trade-off였다 — 당시 환경에서는 전자를 우선한 것이 합리적이었다.

6.4 Modern Challenge — Real-time Decision 요구와의 충돌

영역충돌 지점
Intra-day Visibility 경영진이 당일 실시간 수치를 요구하지만, Batch 모델은 익일 데이터만 제공
24/7 Global 운영 여러 시간대에서 동시에 운영되는 기업은 "업무 외 시간"이라는 개념 자체가 성립하기 어려움
Job Dependency Fragility Batch Job이 순차적으로 연결된 경우, 하나가 실패하면 이후 Job 전체가 지연됨
Architecture Misconception Warning Batch Processing은 낡거나 열등한 처리 방식이 아니다. 제한된 Computing Resource 아래 대규모 마감·평가 작업을 안정적으로 처리하기 위한 합리적인 Scheduling Strategy였다. 문제는 Batch 모델 자체의 결함이 아니라, 24/7 Real-time Enterprise 요구가 "업무 외 시간"이라는 전제를 무너뜨렸다는 점이다.

6.5 S/4HANA Evolution — In-Memory Real-time Calculation

HANA의 Columnar In-Memory 구조는 많은 집계 작업을 야간 Batch 없이 조회 시점에 즉시 계산할 수 있게 했다. 이로 인해 과거 Batch로 미리 계산해 두던 일부 Summary/Aggregate Table 갱신 Job은 더 이상 필요하지 않게 되었다.

다만 이것이 "Batch의 완전한 소멸"을 의미하지는 않는다. 법적·회계적으로 특정 시점에 확정되어야 하는 기간말 마감(Period-end Closing) 절차나, 대량 데이터의 일괄 처리가 더 효율적인 영역은 여전히 Batch/ Scheduled Job으로 운영된다. 변화의 핵심은 "계산을 미리 해둘 필요가 있는 영역"과 "지금 바로 계산해도 되는 영역"을 구분해서, 후자를 실시간으로 옮긴 것이다.
