PART 4. ABAP Programming Legacy — Programming Paradigm Axis

Function Group & Global Variable

LA01~LA06이 "SAP는 왜 이렇게 설계되었는가"를 다뤘다면, LA07부터는 그 결정이 개발자가 코드를 작성하는 방식에 남긴 흔적을 다룬다. 핵심 질문은 하나다 — 왜 SAP는 Class/Object보다 Function Group과 Function Module 중심으로 발전했는가?
Origin Architecture Decision
Early Enterprise Programming Model Constraint → Procedural ABAP Programming Model

7.1 Legacy Origin — Procedural Programming과 Modularization

초기 ABAP은 절차형(Procedural) 언어였다. Class/Object 개념(ABAP Objects)은 훨씬 이후에 도입되었고, 그 전까지 개발자가 로직을 재사용 가능한 단위로 나누는 방법은 Object가 아니라 Function Module 이었다. 여러 개의 Function Module을 하나의 단위로 묶어 관리하는 컨테이너가 Function Group이다.

7.2 Technical Mechanism — Function Group / TOP Include / Global Data

Function Group (Container)
TOP Include — Global Data 선언 영역
Function Module 1
Function Module 2
Function Module 3

Function Group은 TOP Include라는 공통 영역을 가지며, 여기 선언된 변수는 그 Function Group에 속한 모든 Function Module이 파라미터 전달 없이 공유해서 읽고 쓸 수 있다.

" TOP Include (예: SAPLZFG_EXAMPLE)
DATA: gv_counter TYPE i.       " Function Group 전체가 공유하는 Global Data

FUNCTION Z_FM_INITIALIZE.
  gv_counter = 0.
ENDFUNCTION.

FUNCTION Z_FM_INCREMENT.
  gv_counter = gv_counter + 1.   " 파라미터 없이 Global Data 직접 참조
ENDFUNCTION.

7.3 Why It Worked — 당시 환경에서의 합리성

이 구조는 실수가 아니라 당시 SAP Dialog Transaction 처리 모델과 잘 맞았다. 하나의 사용자 세션 안에서 여러 Function Module을 순차 호출할 때, 매번 대용량 데이터를 파라미터로 주고받는 대신 Shared Memory Context(Function Group의 Global Data)를 통해 상태를 공유하면 처리 효율이 높았다. 제한된 Memory와 Roll Area 안에서 동작해야 했던 Dialog Transaction 환경에서는 합리적인 선택이었다. 즉 Explicit Data Passing에서 Implicit Shared Context로의 이동이 이 시기 Architecture Decision의 본질이다.

7.4 Modern Debt — Side Effect와 Hidden Dependency

영역문제
Side Effect Function Module 호출이 Caller가 알 수 없는 Global State를 변경할 수 있음
Hidden Dependency 특정 초기화 Function Module을 먼저 호출해야만 이후 호출이 올바르게 동작 — 호출 순서 의존성이 코드에 명시되지 않음
Testing Difficulty Function Module 하나만 독립적으로 Unit Test하려면 Global State 전체를 함께 설정해야 함
Architecture Misconception Warning Function Group의 Global Variable은 개발자의 실수가 아니다. Object Orientation이 언어 차원에서 지원되지 않던 시대에, 제한된 Memory 환경에서 Modularization을 구현하기 위한 당시의 Programming Model이었다.

7.5 S/4HANA Evolution — ABAP Objects / CDS / RAP

이후 도입된 ABAP Objects는 Class와 Instance 단위로 상태를 캡슐화해, Global Data에 의존하지 않고도 상태를 명시적으로 관리할 수 있게 했다(Encapsulation). CDS는 단순히 "다음 세대 언어 문법"이 아니라 Data Semantics Layer 자체의 진화로, 데이터 모델링 계층에서 의미를 명시한다. RAP(RESTful ABAP Programming Model)는 이 위에서 Business Object를 Behavior Definition으로 캡슐화한다. 전체 방향은 다음과 같다.

Procedural ABAP
        ↓
ABAP Objects        (Encapsulation)
        ↓
CDS                  (Semantic Data Model)
        ↓
RAP                  (Business Object + Behavior)

즉 암묵적 Global State에서 명시적 Instance 기반 State로, 그리고 다시 Semantic Data Model과 Business Object 단위 캡슐화로 이동한 것이다.

Architecture Reading Point LA07은 Programming Paradigm 축의 첫 사례다. 다음 챕터인 LA08(Internal Table Header Line)은 같은 축에서 "Memory Model Simplification"이 어떻게 암묵적 데이터 컨텍스트로 이어졌는지를 다룬다.

One Sentence Takeaway
Function Group과 Global Variable은 초기 ABAP에서 Modularization과 Transaction Efficiency를 확보하기 위한 합리적인 Programming Model이었다. 현대 SAP Architecture는 이를 제거하는 것이 아니라 Explicit Object, Semantic Model, Business Object 기반 구조로 진화시키고 있다.
