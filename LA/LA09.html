PART 4. ABAP Programming Legacy — Programming Paradigm Axis

Dynpro Coupling

LA07은 State를, LA08은 Data Context를 암묵적으로 공유했다. LA09는 그 다음 단계다 — SAP 화면은 왜 Backend Logic과 그토록 강하게 결합되었는가.
Origin Architecture Decision
Network Constraint Management → Integrated Screen Processing Model

9.1 Legacy Origin — Network 비용과 Round Trip 최소화

SAP GUI 시대의 기업 Network는 대역폭이 넉넉하지 않았다. 화면의 필드 하나가 바뀔 때마다 서버에 요청을 보내는 방식은 감당할 수 없는 비용이었다. SAP의 해법은 하나의 화면 전환마다 필요한 로직을 서버 측에서 완결시켜, Round Trip 횟수를 최소화하는 것이었다. 이를 구현한 것이 Dynpro(Dynamic Program)다.

9.2 Technical Mechanism — PBO / PAI Event Model

Dynpro는 Screen Painter로 정의된 화면과, 그 화면에 종속된 Flow Logic으로 구성된다. 화면이 출력되기 전 PBO(Process Before Output)가 데이터를 준비하고, 사용자가 입력한 후 PAI(Process After Input)가 입력값을 검증하고 Backend ABAP Logic을 직접 호출한다.

" Flow Logic (화면에 종속)
PROCESS BEFORE OUTPUT.
  MODULE status_set.        " 화면 표시 전 데이터 준비

PROCESS AFTER INPUT.
  MODULE user_command.      " 입력 검증 + Backend Logic 호출

이 구조에서 화면과 로직은 처음부터 분리를 전제하지 않고 설계되었다 — 하나의 화면 전환을 위한 모든 준비·검증·처리가 서버에서 한 번에 완결된다. PBO/PAI라는 이벤트 구조 자체가 문제라기보다, Screen Event와 Business Logic이 동일한 Program Context 안에 존재했다는 점이 현대 Architecture와의 차이를 만든다.

9.3 Why It Worked — 기업 WAN 환경에서의 효율

PBO/PAI 구조는 여러 개의 개별 요청 대신, 화면 하나당 잘 조율된 하나의 Round Trip으로 동작했다. 당시 기업 WAN 환경에서는 이것이 응답성과 안정성을 모두 확보하는 합리적인 방식이었다. 사용자가 화면 전체를 채운 뒤 한 번에 Submit하는 방식은, Real-time으로 매 입력마다 통신하는 현대 Web 방식보다 당시 조건에서는 오히려 더 효율적이었다.

9.4 Modern Debt — UI와 Logic 분리의 어려움

영역문제
Logic Reuse Backend Logic이 화면 이벤트(PBO/PAI)에 묶여 있어, 화면 없이 동일 로직을 독립적으로 재사용하기 어려움
API 노출 같은 비즈니스 로직을 외부 API로 제공하려면 화면 종속 로직을 별도로 재작성해야 하는 경우가 많음
BDC(Batch Data Communication) 인터페이스가 화면 필드 이름·순서를 그대로 흉내내는 방식으로 동작해, 화면이 바뀌면 인터페이스도 함께 깨짐
Architecture Misconception Warning Dynpro Coupling은 실패한 UI 설계가 아니다. 제한된 Network 환경에서 Transaction Efficiency를 확보하기 위한 합리적인 Interaction Model이었다. 현대 Architecture의 과제는 Dynpro를 부정하는 것이 아니라, UI와 Business Logic 사이의 Boundary를 재정의하는 것이다.

Fiori/UI5는 Presentation을 별도 계층으로 분리하고, OData Service는 Business Logic을 화면과 무관하게 호출 가능한 형태로 노출한다. RAP의 Behavior Definition은 Validation·Transaction을 포함한 하나의 Business Object 로직으로 캡슐화되고, 그 아래 CDS Data Model이 Semantic Layer를 담당한다. 이 구조 덕분에 Fiori, 모바일 앱, 외부 API 등 여러 Consumer가 동일한 Business Object를 동시에 소비할 수 있다.

Architecture Reading Point LA09는 Programming Paradigm 축에서 "UI Coupling" 사례다. 다음 챕터인 LA10(T-Code 중심 UX)는 같은 축에서 "사용자가 시스템을 배우는 방식" 자체가 어떻게 Architecture의 일부가 되었는지를 다룬다.

One Sentence Takeaway
Dynpro는 Network Constraint 시대에 Transaction Efficiency를 확보하기 위한 Integrated Processing Model이었다. 현대 SAP Architecture는 이를 폐기하는 것이 아니라 UI, Service, Business Logic의 Boundary를 분리하는 방향으로 진화하였다.
