PART 5. Modern Enterprise Architecture — Legacy Reinterpreted Axis

Digital Enterprise Architecture

새로운 기술은 없다. LA13~LA18에서 쌓아온 여섯 가지 원칙을 하나의 관점으로 통합하는 장이다. 질문은 하나다 — 모든 Capability가 독립적으로 진화한다면, 무엇이 여전히 이것을 하나의 Enterprise로 만드는가?

19.1 Modern Mechanism — Capability Map

LA18은 Capability가 배치될 위치를 다뤘다. LA19의 질문은 그 여러 위치에 흩어진 Capability를 어떻게 하나로 바라보는가다. 답은 System Map이 아니라 Capability Map이다.

핵심은 하나의 시스템이 모든 기능을 제공하는 것이 아니라, 여러 Platform이 함께 하나의 Enterprise Capability를 구현한다는 것이다.

최적화의 단위 자체가 이동했다 — System → Capability → Enterprise. LA13~LA18에서 각 챕터가 개별적으로 다뤘던 최적화(Data 모델, Boundary, Contract, Object, Event, Platform)는 결국 Enterprise 전체 관점에서 조율되어야 의미를 갖는다.

19.3 New Trade-offs — Enterprise 수준의 조율 비용

영역Trade-off
Platform Coordination 여러 Platform에 흩어진 Capability가 서로 어긋나지 않도록 지속적으로 조율해야 함
Enterprise Governance LA18의 Governance가 System 수준이었다면, 이제는 Enterprise 수준의 거버넌스가 필요함
Shared Architecture Standards 각 Platform이 독립적으로 진화하더라도, 공통으로 따라야 할 Architecture 원칙은 공유되어야 함
Capability Ownership 어떤 조직/팀이 어떤 Capability를 소유하고 책임지는지 Enterprise 전체에서 명확해야 함
Architecture Misconception Warning Digital Enterprise is not a collection of connected systems.

Digital Enterprise는 연결된 시스템의 집합이 아니다. It is a collection of coordinated capabilities. — 시스템을 연결하는 것이 아니라, Capability를 조율하는 것이 Digital Enterprise의 본질이다.
Architecture Reading Point LA13(Data) → LA14(Boundary) → LA15(Contract) → LA16(Object) → LA17(Event) → LA18(Platform) → LA19(Enterprise)까지, 추상화 수준이 개별 데이터에서 전체 Enterprise로 완전히 올라왔다. 마지막 챕터인 LA20은 이 일곱 개 개념을 하나의 철학으로 재해석한다.

Architectural Principle
Digital Enterprise는 시스템을 연결하는 것이 아니라 Capability를 조율하는 것이며, 최적화의 단위는 System에서 Enterprise 전체로 올라간다.
Design Law
Optimize the enterprise, not individual systems.
Looking Ahead
If technologies continue to change, what should remain constant? — LA20은 Principle로 이 질문에 답하며 시리즈를 마무리한다.
