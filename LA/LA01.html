PART 1. SAP Legacy Architecture의 탄생 배경

왜 SAP는 이렇게 설계되었는가?

SAP의 오래된 설계 패턴은 임의의 선택이 아니라, 특정 시대의 하드웨어·네트워크·데이터베이스 제약 아래에서 최선의 Engineering 결정이었다. 이 챕터는 그 제약 조건과 설계 철학을 먼저 이해하는 데서 출발한다.

1.1 Enterprise Computing 시대의 제약 조건

SAP의 핵심 설계 패턴 대부분은 1980~90년대 Enterprise Computing 환경의 물리적 제약을 전제로 만들어졌다. 중요한 것은 제약 그 자체가 아니라, 그 제약이 어떤 Architecture Decision으로 이어졌고, 그 결정이 오늘날 어떤 패턴으로 남아 있는지의 연결이다.

당시 제약 Architecture Decision 결과 (Legacy Pattern)
Memory 비용 높음 데이터 중복 최소화 정규화 모델
Network 느림 Application Server 중심 처리 ABAP Logic 집중
DB 성능 제한 Application Layer 계산 LOOP / Internal Table
Storage 제한 Key 압축 Leading Zero
동시 사용자 증가 Logical Isolation MANDT Client
SAP 설계 철학
SAP Legacy Architecture는 "제약을 극복하기 위한 최적화 Architecture"였다.
오늘날 기술 부채로 보이는 대부분의 설계는 당시 Enterprise Computing 환경에서 Transaction Integrity와 Performance를 확보하기 위한 합리적인 Engineering Decision이었다.

1.2 Five Architecture Decisions That Shaped SAP

이 철학은 이후 살펴볼 모든 Legacy 설계 패턴의 공통된 근본 원인이다. 아래 다섯 개의 Architecture Decision은 각각 하나의 문제를 해결하기 위해 내려졌고, 동시에 오늘날 하나의 Legacy Debt로 남아 있다.

Architecture Decision 당시 해결하려던 문제 현재 Legacy Debt
Data Compression Storage 제한 Data Representation Complexity
Application Logic Centralization DB 성능 부족 Code Pushdown 한계
SAP GUI Coupling Network 제한 UI/API 분리 어려움
Strong Transaction Control Enterprise 정합성 Real-time Integration 제약
Proprietary Framework 빠른 ERP 개발 Open Architecture 전환 비용
Architecture Reading Map LA01은 Legacy 기능을 설명하는 챕터가 아니다. 이후 등장하는 모든 Technical Debt는 위 다섯 가지 Architecture Decision의 결과다. LA02부터는 이 표의 각 행이 하나씩 실제 Legacy Pattern 사례로 펼쳐진다.
