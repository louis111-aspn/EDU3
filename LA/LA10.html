PART 4. ABAP Programming Legacy — Programming Paradigm Axis

T-Code 중심 UX

LA07~LA09가 코드와 화면 내부의 결합을 다뤘다면, LA10은 사용자 경험 자체로 확장된다. 왜 SAP 사용자는 화면을 "찾는" 것이 아니라 Transaction을 "실행"하는 방식으로 시스템을 학습했는가?
Origin Architecture Decision
Command Efficiency → Transaction Code Navigation

10.1 Legacy Origin — Command 기반 Navigation의 배경

LA09에서 본 것처럼, 화면 전환 자체가 Network 비용을 수반하던 시절 메뉴를 여러 단계 눌러 들어가는 방식은 비효율적이었다. SAP는 대신 4자리 코드(T-Code)를 입력하면 어떤 메뉴 depth에 있든 즉시 목표 Transaction으로 이동하는 방식을 제공했다. 반복적으로 같은 업무를 수행하는 사용자에게는 메뉴 탐색보다 Command 입력이 훨씬 빨랐다.

10.2 Technical Mechanism — T-Code와 TSTC

모든 T-Code는 TSTC Table에 등록되어 특정 Program/Screen과 매핑된다. 사용자는 GUI 상단 Command Field에 T-Code를 입력하는 것만으로 현재 화면과 무관하게 목표 Transaction으로 직접 이동한다.

" Command Field 입력 예시
/nVA01     " 현재 세션 종료 후 새 Transaction(영업 오더 생성) 시작
/oVA01     " 새 세션(창)에서 Transaction 시작
Important Concept Transaction Code는 화면 번호가 아니라 Application Transaction Entry Point다. 초급자는 종종 "VA01 = 특정 화면"이라고 이해하지만, 실제로는 Transaction Code → Transaction Definition → Program/Screen/Object로 이어지는 매핑의 시작점일 뿐이다.

매일 같은 Transaction을 반복 실행하는 현업 사용자(영업 담당자, 회계 담당자 등)에게 T-Code는 일종의 단축키였다. Role이나 메뉴 구조와 무관하게 동일한 진입점을 제공한다는 점도 일관성 측면에서 장점이었다.

10.4 Modern Debt — 신규 사용자 학습 곡선

영역문제
Onboarding 난이도 VA01 자체는 "영업 오더 생성"이라는 업무 의미를 전혀 담고 있지 않음 — 코드를 암기해야 접근 가능
UX Fragmentation 수백 개 모듈에 걸친 T-Code를 개별적으로 암기해야 하며, Role에 맞는 것만 걸러서 보여주는 구조가 아님
Business Process 비가시성 T-Code는 단일 Transaction 진입점 단위로 존재해, 여러 Transaction에 걸친 End-to-End 프로세스 흐름 자체를 화면 구조가 보여주지 못함
Architecture Misconception Warning T-Code 중심 UX는 사용자를 배려하지 않은 설계가 아니다. 반복 업무를 수행하는 숙련 사용자의 생산성을 극대화하기 위한 Command-oriented Interaction Model이었다. 문제는 신규 사용자, Self-service, Role-based Experience가 중요해진 시대에 학습 방식과 업무 발견성(Discoverability)이 제한되었다는 점이다.

10.5 Modern Evolution — Role-based Fiori Experience

Fiori Launchpad는 T-Code가 아니라 Business Role을 진입점으로 삼는다. 사용자는 자신의 Role에 할당된 Business Catalog를 통해 관련 Fiori App만 타일 형태로 마주하게 된다. 다만 이는 T-Code 체계를 완전히 없앤 것이 아니다 — 상당수 Fiori App은 내부적으로 여전히 기존 Transaction/OData Service를 호출하며, T-Code는 숙련 사용자를 위한 빠른 진입 수단으로 병행 지원되는 경우가 많다.

즉 Fiori는 "화면을 교체"한 것이 아니라, Business Intent 중심의 진입점을 추가하고 기존 Transaction은 Compatibility Layer로 유지하는 방식으로 진화했다 — LA09에서 본 RAP Business Object 구조가 이 Modern Entry Point의 기반이 된다.

Architecture Reading Point LA10은 User Interaction 축의 사례다. 다음 챕터인 LA11(Modification Culture)는 축이 다시 바뀐다 — "확장 방식(Extension Model)"이라는 SAP Technical Debt의 가장 큰 원인을 다룬다.

One Sentence Takeaway
T-Code 중심 UX는 Enterprise Transaction Efficiency를 위해 탄생한 Command-oriented Interaction Model이었다. 현대 SAP Architecture는 이를 제거하는 것이 아니라 Business Role과 User Intent 중심 Experience로 확장하고 있다.
