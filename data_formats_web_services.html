CHAPTER 1

왜 컨설턴트가 이 배경을 알아야 하는가

"포맷 선택"이 아니라 "시대적 설계 철학의 산물"이라는 관점에서 접근합니다.

프로젝트 인터페이스 설계 회의에서 "SOAP로 갈까요, OData로 갈까요?"라는 질문은 실무에서 반복적으로 등장합니다. 그런데 이 질문에 답하기 위해서는 먼저 SOAP와 OData가 정확히 무엇을 비교하는 대상인지부터 짚어야 합니다.

실무에서 반복되는 세 가지 질문

  • 레거시 ECC 인터페이스는 왜 대부분 SOAP 기반인가?
  • S/4HANA, Fiori, SAC는 왜 OData를 표준으로 채택했는가?
  • "SOAP는 XML, REST는 JSON"이라는 이해는 왜 정확하지 않은가?
일반화 주의 이 모듈에서 다루는 SOAP vs OData 비교는 "어느 것이 더 나은가"가 아니라 "왜 서로 다른 시대에 서로 다른 목적으로 설계되었는가"를 이해하기 위한 것입니다. 실제 프로젝트에서의 인터페이스 방식 선택은 고객 시스템 버전, 기존 인터페이스 자산, 유지보수 조직 역량 등 프로젝트별 설계 결정 사항입니다.

SGML — 모든 마크업 언어의 공통 조상

SGML(Standard Generalized Markup Language)은 1986년 ISO 표준으로 제정된, 문서에 구조를 부여하기 위한 메타 언어입니다. SGML 자체가 특정 언어라기보다는 "태그로 구조를 표시하는 규칙의 집합"이었고, 이 틀 안에서 이후 HTML과 XML이 각각 파생되었습니다.

오해 HTML과 XML은 태그 문법이 비슷하므로 결국 같은 목적의 기술이다.
사실 HTML은 문서를 브라우저에 렌더링하기 위한 언어이고, XML은 데이터를 시스템 간에 교환하기 위한 언어입니다. 목적 자체가 다릅니다.
근거 HTML은 <table>, <div>처럼 정해진 태그 집합만 사용하며 시각적 표현을 목표로 합니다. 반면 XML은 태그 이름을 개발자가 자유롭게 정의(<InvoiceAmount> 등)하며, 렌더링이 아니라 구조화된 데이터 전달이 목표입니다.
결론 같은 SGML 계보에서 갈라졌지만, HTML은 "보여주기 위한 언어", XML은 "교환하기 위한 언어"로 목적이 분화되었습니다.

JSON의 등장 예고

2000년대 들어 웹 브라우저에서 JavaScript가 비동기로 서버와 데이터를 주고받는 방식(AJAX)이 확산되면서, XML보다 가볍고 JavaScript가 네이티브로 처리할 수 있는 포맷에 대한 수요가 커졌습니다. 이 배경은 4장에서 자세히 다룹니다.

Q. SGML은 지금도 사용되는가?
SGML 자체는 현재 신규 시스템에서 거의 사용되지 않습니다. 다만 HTML과 XML이라는 두 개의 핵심 후속 언어를 통해 그 설계 철학(태그 기반 구조화)은 현재까지 이어지고 있습니다.

CHAPTER 3

XML 심화 — 왜 엔터프라이즈가 XML을 선택했는가

엔터프라이즈 시스템이 데이터 교환 포맷에 요구했던 속성과, XML이 이를 어떻게 충족했는지 살펴봅니다.

Well-formed vs Valid

구분정의예시 관점
Well-formedXML 문법 규칙(태그 열고 닫기, 중첩 규칙 등)을 지킨 상태구조적으로 "깨지지 않은" 문서
ValidWell-formed일 뿐 아니라, 특정 Schema(XSD/DTD)가 정의한 규칙까지 만족한 상태비즈니스 규칙까지 검증된 문서
핵심 구분 Well-formed는 "문법적으로 맞는가", Valid는 "약속한 계약(Schema)을 지켰는가"를 의미합니다. 엔터프라이즈 연동에서는 Valid 여부, 즉 Schema 검증이 훨씬 중요한 의미를 가집니다.

Schema(XSD) — 데이터 계약의 근거

XSD(XML Schema Definition)는 XML 문서가 어떤 요소를 포함해야 하는지, 각 요소의 데이터 타입은 무엇인지, 필수/선택 여부는 무엇인지를 명시적으로 정의합니다. 두 시스템이 인터페이스를 구축할 때 이 XSD 자체가 "사전에 합의된 데이터 계약서" 역할을 합니다.

Self-describing과 Namespace

  • Self-describing: 태그 이름 자체가 데이터의 의미를 설명 (<CompanyCode>는 그 자체로 의미 전달)
  • Namespace: 서로 다른 시스템에서 정의한 동일한 이름의 태그가 충돌하지 않도록 구분하는 메커니즘

엔터프라이즈가 원했던 속성 → XML이 제공한 답

검증 가능한 데이터 계약이 필요하다
↓
XSD를 통한 Schema 검증
시스템 간 태그 충돌을 막아야 한다
↓
Namespace를 통한 구분

이러한 속성들 — 검증 가능성, Self-describing, Namespace — 은 이후 5·6장에서 다룰 "메시지 포맷으로서의 XML 채택 근거"로 다시 연결됩니다. 지금은 이 속성들 자체를 XML의 특성으로만 이해하면 충분합니다.

CHAPTER 4

JSON 심화 — 왜 XML의 자리를 밀어냈는가

웹/모바일 생태계의 요구가 어떻게 더 가벼운 데이터 표현 방식을 필요로 했는지 살펴봅니다.

AJAX와 브라우저 생태계의 등장

2000년대 중반, 웹 페이지 전체를 새로고침하지 않고 서버와 부분적으로 데이터를 주고받는 AJAX(Asynchronous JavaScript and XML) 방식이 확산되었습니다. 역설적으로 AJAX의 이름에는 XML이 들어가지만, 실제로는 JavaScript가 네이티브로 파싱할 수 있는 더 가벼운 포맷에 대한 요구가 커지면서 JSON(JavaScript Object Notation)이 그 자리를 대체하게 되었습니다.

비교 항목XMLJSON
파싱 방식별도 파서 필요JavaScript 엔진에서 네이티브 처리
문서 크기태그 반복으로 상대적으로 큼경량 (Key-Value 구조)
스키마 검증XSD를 통한 강한 검증표준 스키마 부재 (JSON Schema는 후발 보완)
가독성태그 구조로 다소 장황간결하고 직관적
오해 JSON이 등장한 이후로 XML은 구식 포맷이 되었으며, JSON이 XML보다 무조건 우월하다.
사실 JSON은 가볍고 빠른 처리에 유리하지만, 계약 기반의 엄격한 검증이 필요한 영역에서는 XML의 XSD 검증이 여전히 더 신뢰할 수 있는 선택입니다.
근거 JSON에는 XSD와 동등한 수준으로 명세에 내장된 표준 스키마가 없습니다. JSON Schema라는 보완 표준이 있지만 별도로 채택해야 하는 부가 기술이며, XML처럼 처음부터 강제되지 않습니다. B2B EDI, 금융권 표준 메시지처럼 데이터 무결성이 중요한 영역에서는 XML이 여전히 표준으로 남아 있습니다.
결론 JSON과 XML은 "구식 vs 신식"의 우열 관계가 아니라, 속도·경량성이 우선인 영역(JSON)과 검증·계약이 우선인 영역(XML)으로 용도가 나뉜 관계입니다.
Q. JSON에는 정말 스키마가 없는가?
JSON Schema라는 보완 표준이 존재하지만, XSD처럼 XML 명세 자체에 내장된 표준이 아니라 별도로 채택해야 하는 부가 기술입니다. 이 때문에 "기본적으로 스키마 강제가 없다"는 JSON의 특성 자체는 유효합니다.

CHAPTER 5

데이터를 주고받는 방법의 진화

이번 모듈에서 가장 중요한 연결고리입니다. 지금까지는 "표현 방식(포맷)"만 다뤘다면, 여기서부터는 "전달 방식(프로토콜/아키텍처)"을 다룹니다.

오해 SOAP는 XML과, REST는 JSON과 짝을 이루는 개념이다.
사실 SOAP는 XML 위에서 동작하는 메시징 프로토콜이고, REST는 HTTP를 사용하는 아키텍처 스타일입니다. 이 둘은 "전달 방식" 층위의 개념이며, XML·JSON이라는 "표현 방식" 층위와는 서로 다른 차원입니다.
근거 SOAP 메시지는 반드시 XML로 작성되지만, REST 기반 통신은 JSON뿐 아니라 XML로도 표현될 수 있습니다(실제로 초기 REST API 중에는 XML 응답을 제공하는 경우도 있었습니다). 즉 REST가 곧 JSON을 의미하지는 않습니다.
결론 3·4장에서 다룬 XML/JSON은 "표현(포맷)" 층위, 이번 장 이후 다룰 SOAP/REST는 "전달(프로토콜/아키텍처)" 층위입니다. 두 층위를 1:1로 대응시키지 않는 것이 이 모듈 전체를 관통하는 핵심 원칙입니다.

데이터가 시스템 간에 이동하기까지

Data — 표현된 정보(XML/JSON)
↓
File — 저장 단위로 존재
↓
Network — 시스템 간 물리적 연결
↓
HTTP — 웹의 표준 통신 규약
↓
Web Service — 시스템 간 기능 호출

Protocol vs Architecture Style

Web Service 단계에서 실제로 "어떤 규칙으로 통신할 것인가"를 결정하는 두 가지 큰 갈래가 등장합니다.

SOAP

엄격한 메시지 형식을 정의하는 프로토콜(Protocol)

REST

HTTP를 활용하는 설계 원칙, 즉 아키텍처 스타일(Architecture Style)

이 둘의 심화 내용은 각각 6장과 7장에서 다룹니다. 이 장에서 기억해야 할 것은 단 하나 — SOAP와 REST는 "같은 층위의 선택지"이지, 포맷(XML/JSON)과 직접 짝을 이루는 개념이 아니라는 점입니다.

CHAPTER 6

SOAP와 XML

SOAP가 왜 XML을 메시지 포맷으로 선택했는지, 그리고 엔터프라이즈 통합에서 어떻게 표준으로 자리잡았는지 살펴봅니다.

SOAP의 핵심 구성 요소

Envelope
SOAP 메시지 전체를 감싸는 최상위 XML 요소
WSDL
서비스가 제공하는 Operation, 메시지 구조, 접속 정보를 정의하는 계약 문서
Operation
서비스가 제공하는 개별 기능 단위(예: CreateInvoice)
Binding
Operation을 실제 통신 프로토콜(HTTP 등)에 연결하는 방식

왜 SOAP는 XML을 선택했는가

SOAP
↓ 메시지 포맷으로
XML
↓ 계약 검증을 위해
XSD
↓ 설계 방식
Contract First

SOAP는 3장에서 다룬 XML의 속성 — 검증 가능성(XSD), Self-describing, Namespace를 통한 충돌 방지 — 을 그대로 활용합니다. 서비스를 만들기 전에 WSDL(계약)을 먼저 정의하고, 그 계약에 따라 클라이언트와 서버가 개발되는 방식을 Contract First라고 부릅니다.

ECC 시대의 표준 ECC 시대의 SAP 인터페이스 설계에서 SOAP가 표준으로 자리잡았던 이유는, 엄격한 계약(WSDL)과 강한 타입 검증이 당시 엔터프라이즈 시스템 간 연동에서 요구되던 안정성과 맞아떨어졌기 때문입니다.

CHAPTER 7

REST와 OData

REST 원칙이 무엇인지, 그리고 OData가 이 원칙 위에 어떻게 SAP 표준으로 자리잡았는지 살펴봅니다.

REST의 핵심 원칙

  • Resource: 데이터를 URI로 식별되는 자원으로 취급 (예: /Invoices/1000)
  • HTTP Method: GET(조회), POST(생성), PATCH(수정), DELETE(삭제)로 행위를 표현
  • Stateless: 각 요청은 이전 요청 상태에 의존하지 않음

OData — REST 위에 세워진 표준

REST — 아키텍처 원칙
↓ 원칙 위에 메타데이터·쿼리 옵션 추가
OData — Application Protocol
↓ SAP 생태계에서
SAP API (Fiori, SAC Write-back 등)
오해 OData는 REST와 사실상 같은 것이다.
사실 OData는 REST의 부분집합이 아니라, REST 원칙 위에 메타데이터 문서, 표준화된 쿼리 옵션, 표준 응답 구조를 추가로 정의한 Application Protocol입니다.
근거 REST는 Resource·HTTP Method·Stateless라는 원칙만 제시할 뿐, 쿼리 방법이나 메타데이터 형식까지는 규정하지 않습니다. OData는 여기에 $filter, $expand 같은 표준 쿼리 옵션과 메타데이터 문서를 추가로 정의하여, 같은 REST 원칙을 따르는 API라도 시스템마다 제각각이던 조회 방식을 표준화했습니다.
결론 모든 OData 서비스는 REST 원칙을 따르지만, 모든 REST API가 OData인 것은 아닙니다. OData는 REST라는 큰 원칙 위에 세워진 하나의 구체적인 표준입니다.

SOAP가 WSDL이라는 계약 문서를 갖듯, OData는 메타데이터 문서(Metadata Document)를 통해 서비스가 제공하는 Entity, 필드, 관계를 기술합니다. 다만 WSDL과 달리 REST 원칙을 따르므로 HTTP Method와 URI 기반으로 훨씬 가볍게 접근할 수 있습니다.

CHAPTER 8

SAP 인터페이스 기술의 진화

지금까지 다룬 개념들이 SAP 생태계에서 실제로 어떻게 구현되었는지 정리합니다. SAP 전체 기술이 아니라, 인터페이스 기술의 계보임에 유의하십시오.

← 좌우로 스크롤하면 전체 구조를 확인할 수 있습니다 →
오해 SAP_COM_0065(SAC Live Connection 통신 시나리오)는 곧 InA 프로토콜이다.
사실 InA(Information Access)는 프로토콜이고, SAP_COM_0065는 Communication Scenario, 즉 특정 통신 목적을 위해 패키징된 인증·인터페이스 묶음입니다. 서로 다른 층위의 개념입니다.
근거 SAC Live Connection이 S/4HANA와 통신할 때 SAP_COM_0065 시나리오가 활성화되고, 그 내부에서 실제 데이터 조회는 InA protocol을 통해 이뤄집니다. 즉 SAP_COM_0065는 "이 통신을 가능하게 하는 설정 패키지"이고, InA는 "그 안에서 실제로 사용되는 프로토콜"입니다. 구체적인 활성화·인증 절차는 이번 모듈의 범위를 벗어나므로 다루지 않습니다.
결론 Communication Scenario ≠ Protocol. 이 구분을 명확히 해야 SAP 인터페이스 설계 문서를 정확히 읽을 수 있습니다.
일반화 주의 이 계보도는 대표적인 흐름을 보여주는 것으로, 실제 프로젝트에서는 고객사의 시스템 버전, 레거시 인터페이스 자산 현황에 따라 SOAP Web Service, RFC 기반 연동이 여전히 병행되는 경우가 많습니다. "OData가 SOAP를 완전히 대체했다"는 이해는 정확하지 않습니다.

CHAPTER 9

다섯 개 층위 — 용어가 뒤섞여 보이는 이유

이 모듈 전체의 최종 수렴 지점입니다. 지금까지 등장한 모든 개념을 서로 다른 층위로 분류하여 정리합니다.

구분역할예시
Markup Language문서/데이터에 구조를 태그로 표시HTML, XML
Serialization Format데이터를 표현XML, JSON
Transport Protocol데이터가 네트워크로 이동하는 기본 규약HTTP
Messaging Protocol메시지 형식·계약을 엄격히 규정SOAP
Architecture StyleAPI 설계 원칙REST
API / Application ProtocolREST 기반 데이터 접근 규격OData
SAP-specific ProtocolSAP 분석용 통신 규격InA
SAP Technology인터페이스를 실제로 구현하는 기술 요소RFC, BAPI, Gateway, CDS, CAP
SAP Configuration Artifact통신 목적을 위해 패키징된 설정 단위 (프로토콜이 아님)Communication Scenario (SAP_COM_00XX)
이 모듈의 핵심 학습 포인트 "SOAP vs REST", "XML vs JSON", "SOAP Web Service vs OData"는 서로 다른 세 개의 비교축입니다. 이 표 하나만 정확히 이해하면 향후 CDS View, OData Service, Integration Suite, SAP Gateway, SAC Live Connection, CAP 등을 학습할 때 개념이 흔들리지 않습니다.
  • XML과 JSON은 표현(Serialization) 층위이며, 서로 경쟁 관계다.
  • SOAP와 REST는 전달(Protocol/Architecture) 층위이며, 서로 경쟁 관계다.
  • OData는 REST 위에 세워진 Application Protocol이지, REST 자체와 동일하지 않다.
  • InA는 OData와 별개의 SAP 전용 프로토콜이다.
  • Communication Scenario(SAP_COM_00XX)는 프로토콜이 아니라 설정 단위(Configuration Artifact)다.
